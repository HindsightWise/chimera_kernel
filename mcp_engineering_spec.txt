# Chimera Kernel: Model Context Protocol (MCP) & Autonomous Forge Subsystem
**Engineering Specification Document v1.0**

## Executive Summary
This document outlines the architectural integration of Anthropic's **Model Context Protocol (MCP)** into the native Rust substrate of the **Chimera Kernel**, alongside the implementation of an **Autonomous Forge (The Maker Capability)**.

This upgrade transitions the Kernel from utilizing a static, hard-coded Rust toolchain to acting as a dynamic, polymorphic agent orchestrator capable of consuming external capability schemas in real-time and even coding its own persistent tools without rebooting.

---

## 1. The MCP Gateway (Native Rust Subsystem)

### 1.1 Structural Overview
The MCP specification defines a standard JSON-RPC 2.0 communication format for AI tools across standard input/output (Stdio) streams. Because Chimera is a strictly asynchronous Rust application (Tokio), proxying this through external Python bridging creates unacceptable latency and memory fragmentation.

To solve this, we implemented `src/architecture/mcp_gateway.rs`. 

**Responsibilities of the `McpGateway`:**
- **Process Management**: Spawns isolated subprocesses (`tokio::process::Command`) targeting external runtimes (e.g., Node.js via `npx`, Python via `uvx`).
- **I/O Parsing**: Attaches non-blocking `BufReader` streams directly to the `stdin`/`stdout` of the spawned processes.
- **Protocol Handshake**: Manages the JSON-RPC state machine (`initialize` -> `notifications/initialized` -> `tools/list` -> `tools/call`).
- **Schema Mapping**: Ingests `inputSchema` properties from the MCP Server and structurally maps them to `async_openai::types::ChatCompletionTool`.

### 1.3 Asynchronous Mutex & Lifetime Design
The `McpGateway` is initialized prior to the main `loop {}` inside `agent.rs` and bound to an `Arc<McpGateway>`. The gateway utilizes interior mutability (`RwLock` for fast read access to capability strings, and `Mutex` for asynchronous message callbacks mapping `JSON-RPC id` -> `oneshot::Sender`). This guarantees thread-safety and eliminates Tokio blocking issues when the LLM stalls on a long-running external process.

## 2. Dynamic Tool Ingestion Pipeline

### 2.1 The `mcp_registry.json` Source of Truth
The gateway operates strictly off a whitelist root configuration located at `mcp_registry.json`.

```json
{
  "servers": {
    "sequential_thinking": {
      "command": "/opt/homebrew/bin/npx",
      "args": ["-y", "@modelcontextprotocol/server-sequential-thinking"],
      "env": {}
    }
  }
}
```

### 2.2 Boot Sequence Integration
During boot sequence, `load_servers()` spawns the processes parallel to the kernel. As they report back their tool schemas (via `tools/list`), those schemas are cached natively inside operations memory (`Gateway.schemas`).

When `agent.rs` invokes the deepseek API, `tools::get_tools(mcp_gateway.clone()).await` appends this living schema array into the OpenAI request object concurrently, allowing the AI seamless visibility of any tools that resolve their handshake.

## 3. The Autonomous Forge (Self-Coding Capability)

We implemented a native Rust tool called `forge_mcp_server` (`src/tools/forge.rs`). This tool grants the AI autonomous capabilities to recursively expand its own operational environment.

### 3.1 Forge Operation Lifecycle
1. **The Gap**: The LLM encounters a specific logical gap where it needs a tool it does not currently possess.
2. **The Request**: It invokes `forge_mcp_server` targeting a Node.js execution environment, passing raw Javascript logic inside `javascript_code` alongside a required JSON variable dictionary mapping `parameters_schema`.
3. **The Boilerplate**: The Rust kernel surrounds the LLM's raw javascript with the official `@modelcontextprotocol/sdk` boilerplate code (handling structural definitions, schema bindings, and Stdio setup).
4. **File Output**: The resulting Javascript file is flushed to disk at `src/mcp_servers/{skill_name}.js` with `0o755` executable permissions.
5. **Registry Binding**: The internal `mcp_registry.json` is updated to include the new script path, ensuring the new tool persists across system reboots.
6. **Hot-Swap Loading**: The `forge` executes a direct system ping (`spawn_server_from_value`) back into the `McpGateway`, instantly launching and ingesting the new Node.js server. The new tool is available globally in the very next inference tick.

## 4. Operational Resilience and Future-Proofing

- **Agnostic Runtimes**: Because the JSON-RPC interface operates over generic `stdin`/`stdout`, the architecture natively supports any MCP server language (Python/uvx, Go, Node.js) provided the `mcp_registry.json` specifies the correct bash wrapper. 
- **Graceful Error Handling**: If a spawned Node.js process crashes, the `tools::execute` block catches the dead transmitter and returns a stringified error directly to the LLM agent, allowing the LLM to recursively fix its own runtime logic via additional tool calls.
- **Fail-Safe Integration**: This architecture exists entirely upstream of the `Neural Fail-Safe` protocol, meaning an API cutover to local Ollama models still perfectly preserves the MCP tool array functionality, granting local 4-bit LLaMAS full access to the AI's custom skills.
